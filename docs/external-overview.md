# 法人番号 MCP 外部向け概要

## 1. 要約

会社名は、たいてい少し間違っている。名刺では「(株)」になり、営業リストでは旧字体が混ざり、請求書では全角と半角が並ぶ。それでも、顧客管理、与信、請求、KYC、バックオフィスのデータベースは、正しい法人番号と正式な所在地を必要とする。

`@sugukuru/mcp-houjin-bangou` は、国税庁 法人番号 Web-API Ver.4.0 を AI エージェントから安全に使うための Model Context Protocol (MCP) サーバである。13桁の法人番号検索、法人名検索、チェックデジット検証、表記揺れ正規化、出典文生成を提供し、Tools だけでなく Prompts、Resources、Resource Templates、Completion、Logging、Pagination、UI Resources、Server Card まで実装している。

目的は、単なる API ラッパーを出すことではない。不完全な会社情報を、公的な事実へ接続する小さな標準層を作ることにある。

## 2. 現在できること

- 13桁の法人番号から法人情報を取得する。
- 会社名から国税庁 API のあいまい検索を使って候補を探す。
- API を呼ばずに法人番号のチェックデジットを検証する。
- 国税庁 `target=1` が吸収しない 7 種類の表記揺れを、T7 `normalize_company_name` で補完する。
- 国税庁の利用規約に必要な出典文を、実務出力に添える。
- 名刺 OCR、営業リスト enrichment、顧客マスタ重複検知の MCP Prompts を同梱する。
- `corp://{corporate_number}`、`attribution://houjin-bangou`、UI Resources を通じて、AI クライアントが結果を再利用しやすい形で渡す。

## 3. 実務ワークフロー

```mermaid
flowchart LR
  rawInput["不完全な会社情報"] --> normalize["T7表記揺れ正規化"]
  normalize --> search["国税庁API検索"]
  search --> verify["法人番号と所在地の確認"]
  verify --> enrich["CRM向け補完レコード"]
  enrich --> review["人間の最終確認"]
```

名刺 OCR では、AI が名刺テキストから会社名を拾い、表記揺れを正規化し、国税庁 API で法人番号と所在地を確認し、CRM に入れられる形のレコードを作る。営業リスト enrichment では、Excel や CSV 由来の不完全な企業名を一括で照合する。顧客マスタ重複検知では、法人番号を強いキーとして使い、名前と住所の類似も補助的に見る。

この一連の流れは、派手な AI デモというより、事務作業の中に残っている小さな誤差を取り除く仕事である。人間が国税庁の検索画面を開き、表記を変えながら何度も試していた作業を、AI エージェントが手順として再現できるようにする。

## 4. 日本の MCP 市場の現在地

日本の MCP 市場は、まだ「導入事例が大量に積み上がった成熟市場」ではない。Cursor、Claude Desktop、VS Code 系の開発環境を使う技術者から先に広がり、次に社内業務・SaaS 連携・行政データ接続へ伸びていく段階にある。

現時点での特徴は、次の通りである。

- MCP は「AI に外部ツールを安全に使わせる標準インターフェース」として認知され始めている。
- 日本語圏では、汎用デモよりも、会計、労務、行政手続、営業管理、補助金、入管、金融などの業務データ接続に需要がある。
- 企業導入では、ツールの便利さだけでなく、認証、監査、ログ、出典、権限管理、レート制限、個人情報の扱いが問われる。
- 日本の公的データは有用だが、仕様書、利用規約、文字コード、分割取得、出典表記などの実務上の壁が残る。

この MCP は、その中で「日本の法人データを AI エージェントが扱うための基礎部品」として置ける。市場全体が大きく騒がしくなる前に、仕様書に忠実な実装、テスト、出典、公開パッケージ、ADR を積んでおくことに意味がある。

## 5. ターゲット

主な対象は、B2B 営業、CRM/ERP ベンダー、AI エージェント開発者、会計・請求 SaaS、税理士事務所、与信・KYC・バックオフィス担当者である。特に、手入力、名刺、紹介、展示会リスト、請求書、古い顧客台帳から企業データを作る現場に向いている。

より具体的には、次の層が対象になる。

- **開発者・AI エージェント利用者**: Claude Desktop、Cursor、VS Code などから法人番号検索を自然言語で使いたい人。
- **SaaS 事業者**: CRM、請求、会計、営業支援、与信、KYC に法人番号・所在地補完を組み込みたい企業。
- **バックオフィス部門**: 顧客マスタの重複、法人名の表記揺れ、請求先情報の不備を減らしたい担当者。
- **士業・経理領域**: 請求書、インボイス、法人実在確認、取引先管理を正確にしたい税理士・会計事務所。
- **スグクル自身**: 農業人材派遣、在留資格、請求、取引先管理の実務の中で、公的データを安全に扱うための基盤。

## 6. 競合状況

競合は、特定の OSS だけではない。実際の代替手段は、国税庁 API を直接叩くこと、Web 画面で手検索すること、LLM に推測させること、各社 CRM の内部名寄せ機能、または単純な API ラッパーである。

この MCP の立ち位置は、それらを攻撃することではなく、境界を正しく引くことにある。国税庁 API `target=1` が既に対応している、ひらがな・カタカナ、英大小文字、中点削除は再実装しない。逆に、前株・後株、法人種別、旧字体、NFKC、英語法人名、IVS、空白正規化など、仕様上残る実務のずれだけを T7 で補う。

差別化は、目立つ機能の数ではなく、次の点にある。

- 国税庁仕様を読んだ上で、API が既にやることを重複実装しない。
- 会社名の表記揺れを、LLM の推測ではなく決定的ロジックで補正する。
- すべての出力に出典文を持たせ、利用規約上の責任を残す。
- MCP の Tools だけでなく、Prompts、Resources、Resource Templates、Completion、Logging、Pagination、UI Resources、Server Card まで揃える。
- npm で公開済みで、Claude Desktop / Cursor / VS Code 系の MCP クライアントからすぐ試せる。

## 7. 価値

価値は、検索そのものよりも、その前後にある。人間が何度も直していた会社名を、AI が安全に公的データへ接続できる。チェックデジット検証で無駄な API 呼び出しを減らし、レート制限で公的 API に礼儀正しく接続し、出典文で利用規約への責任を残す。

B2B データにおいて、法人番号は会社名より安定している。正式名称、所在地、法人種別、変更履歴の手がかりを、AI エージェントが扱える形で返すことは、営業、経理、KYC、請求、顧客管理の小さな摩擦を減らす。

組織にとっての価値は、次のように整理できる。

- 顧客マスタ作成の手作業を減らす。
- 取引先の正式名称・所在地確認を標準化する。
- AI エージェントが公的データを使う時の安全な接続点になる。
- 監査・説明・出典のある形でデータ補完を行える。
- 将来のインボイス API、補助金、入管、金融、自治体データ MCP 群への入口になる。

## 8. 収益化の方法

OSS として npm に公開する部分は、入口である。収益化は、法人番号検索そのものを閉じることではなく、運用、統合、責任、継続性の部分で行う。

想定される収益化は次の通りである。

- **Hosted 版**: 自社で MCP サーバを立てられない企業向けに、認証、監査ログ、レート制限、稼働監視付きで提供する。
- **Enterprise Auth**: Enterprise-Managed Authorization、OAuth Client Credentials、IP 制限、組織単位の quota を提供する。
- **SaaS 組み込みライセンス**: CRM、会計、請求、KYC、営業支援 SaaS に法人番号・T 番号補完機能を組み込む。
- **導入支援**: Document AI、社内 DB、Supabase、Cloud SQL、GCP、Slack、Google Sheets、CRM との安全な接続を構築する。
- **業務別 MCP パッケージ群**: 法人番号、インボイス、補助金、入管、金融、自治体データを Sugukuru OSS Lab として連続的に提供する。
- **保守契約**: 国税庁仕様変更、MCP SDK 更新、セキュリティ対応、依存関係更新、監査資料作成を継続的に請ける。

無料 OSS は、信頼を生むための公開された実装である。売るのは検索関数ではなく、壊れにくい運用と、企業データが間違わないための責任である。

## 9. 公開先・配布先の概要

現在の公開・配布先は以下である。

- **npm**: `@sugukuru/mcp-houjin-bangou`
- **GitHub**: `sugukurukabe/mcp-houjin-bangou`
- **GitHub Release**: `v0.2.0`
- **MCP Server Card**: `/.well-known/mcp.json`
- **ドキュメント**: README、ADR、セキュリティ文書、利用規約、プライバシーポリシー、リリースゲート
- **将来の公開先候補**: Zenn 記事、Anthropic Directory、MCP 関連ディレクトリ、Sugukuru コーポレートサイト OSS ページ

公開方針は、露出を目的にしないことである。X や HN で過剰に拡散するより、仕様書、テスト、ベンチマーク、ADR、npm パッケージ、GitHub Release を整える。説明は短く、成果物は重くする。

## 10. ロードマップ

- `v0.2.0`: UI Resources、property-based tests、benchmarks、npm 公開、本番 API live smoke 済み。
- `v0.3.0`: 適格請求書発行事業者 Web-API、T 番号、登録状態検証、請求書リスト一括確認。
- `v0.5.0`: Hosted edition、Enterprise-Managed Authorization、OAuth Client Credentials、運用者向け quota と監査。
- `v1.0.0`: 複数 MCP ホストでの検証、Directory 提出、Skills/Tasks 評価、長期運用の互換性整理。

法人番号の次に T 番号が入ると、B2B KYC、請求、会計、取引先確認の価値は一段上がる。法人が存在するかだけでなく、適格請求書発行事業者として有効かを AI エージェントが確認できるようになるためである。

## 11. 売却の夢と出口

このプロジェクトは、短期的な露出や誇張された市場規模から始めない。まず、小さな公的データ接続を正しく作る。そこから、法人番号、T 番号、補助金、入管、金融、自治体データへ、同じ作法で広げていく。

もし将来、AI エージェントが日本の法人データを扱うとき、自然にこの層を通るようになれば、出口はいくつかあり得る。Hosted 版の企業契約、会計・CRM・金融 SaaS へのライセンス、行政データ MCP 群としての事業譲渡、または Sugukuru OSS Lab 自体の買収である。

ただし、それは売るために飾った資料からは生まれない。仕様書、テスト、出典、運用、そして壊れにくい小さな release の積み重ねからしか生まれない。
