# Spec Reading Checklist — 対象 API 仕様書から抽出すべき 7 観点

本チェックリストは、新規 MCP サーバー着手時に**対象 API の仕様書を読み切る** ための手順書。
本プロジェクトでは国税庁 Web-API 仕様書 (第一編〜第六編) を 8 時間で読破してこのリストを埋めた。

## 7 観点 (すべて埋める)

### 観点 1: 全エンドポイント一覧（バージョン差異含む）

- ベース URL（バージョン path を含むか？）
- エンドポイント名と目的
- HTTP メソッド
- 下位バージョンとの並行稼働ポリシー

**本プロジェクトでの結果**:

```
法人番号 API (Ver.4.0):
- GET /4/num    法人番号指定検索 (最大10件)
- GET /4/diff   取得期間指定検索
- GET /4/name   法人名指定検索

インボイス API (Ver.1.0):
- GET /1/num    登録番号指定検索 (最大10件)
- GET /1/diff   取得期間指定検索
- GET /1/valid  登録番号+日付指定検索 (!← 法人番号APIにはない第3機能)
```

### 観点 2: リクエストパラメータの完全な列挙

各エンドポイントごとに:

- 必須パラメータ（`[必須]` or `[Required]`）
- 任意パラメータ（デフォルト値込み）
- 取り得る値の全列挙（enum の範囲）
- バリデーションルール（桁数・形式・日付範囲等）

特に注意: **複数の似たパラメータの違い**（例: `history` vs `change`, `close` vs `include_closed`）は
必ず「どっちを使うべき状況」を表にする。

### 観点 3: 応答形式と文字コード選択肢

- 選択可能な応答形式（CSV / XML / JSON 等）
- 各形式の type 値
- 文字コード（Shift-JIS / UTF-8 / 両方選択可？）
- エスケープ仕様（CSV の場合 `"` は `""` へ等）
- ヘッダー情報の項目（最終更新日、総件数、分割番号、分割数 等）

**落とし穴例**: 「`type=12` が CSV UTF-8 に見えるが実は XML」等、
公式仕様書を慎重に読まないと間違える対応表は要注意。

### 観点 4: エラーコード一覧

必ず全エラーコードを表にする:

- HTTP ステータスコード
- API 独自エラーコード（400 系の詳細番号）
- エラーメッセージのテキスト
- 発生条件
- 対象エンドポイント（機能別マトリクス）

**本プロジェクトでの実装**: `src/lib/errors.ts` の `NtaErrorCode` と、
Tool output の `error.code` / `error.guidance` に落とし込み。

### 観点 5: レート制限とアクセス制御

明示されている場合:

- RPS 制限 or QPS 制限の数値
- 突破時の HTTP ステータス（通常 403 or 429）
- バックオフ推奨 or Retry-After ヘッダー

明示されていない場合（本プロジェクトの国税庁 API がこれ）:

- 利用規約の「禁止事項」節で「短時間における大量アクセス」等の条項を探す
- 保守的な RPS（1 RPS 等）を設定し、指数バックオフを実装
- 403 受信時のバックオフスケジュール定義

### 観点 6: 利用規約（別添等）の全文

すべての別添を読む（「別添1 利用規約」「別添2 個人情報保護方針」「別添3 サイト利用規約」等）:

- **禁止事項**: 実装で回避すべき全項目
- **出典明記義務**: 正確な指定文言（これが `attribution.service_disclaimer` の原文になる）
- **ID のライフサイクル**: 無利用停止・譲渡禁止・変更届出
- **監査条項**: 国税庁側からの臨場監査・書面監査権限
- **免責条項**: データの正確性について API 提供側の保証範囲

**実装への落とし込み**:

- 禁止事項 → `docs/security/prompt-injection-defense.md` で緩和策公開
- 出典明記 → Zod `AttributionSchema` で全 output に強制
- ID ライフサイクル → Week ly keepalive cron で 3 年無利用停止防止

### 観点 7: 個人情報保護方針

- 提供側が収集する情報の種類
- 利用目的の制限
- 開示請求手続
- 方針変更時の通知方法

**実装への落とし込み**: `docs/policy/privacy-policy.md` で提供側の収集条項と
本プロジェクトの無収集方針を併記。

## 落とし穴検出の 5 つの技法

### 技法 1: 「同じ機能に見える 2 つのパラメータ」を全部リストアップ

例（本プロジェクト）:

- 法人番号 API の `/num?history=0|1` (変更履歴)
  vs `/name?change=0|1` (変更履歴) vs `/name?close=0|1` (登記閉鎖)
- インボイス API の `/num?history=0|1` (登録・取消・失効の履歴) とは**対象が違う**

これらの差を README に載せると「仕様書を深く読む人」と認識される。

### 技法 2: 応答形式の type 値マトリクス

複数 API を扱う場合、同じ type 値が**別の意味を持つ**ことがある:

| API | `01` | `02` | `11` | `12` | `21` |
|---|---|---|---|---|---|
| 法人番号 | CSV/SJIS | CSV/UTF8 | - | XML/UTF8 | - |
| インボイス | CSV | - | XML | - | JSON |

### 技法 3: 内蔵機能を発見する

対象 API が**自分のドメインで自動処理している機能**を見つける:

- 本プロジェクトの例: 国税庁 `/4/name?target=1` が「ひらがな→カタカナ」「英大小文字変換」「中点削除」を自動処理
- 既存実装の多くでは、この仕様を前提にしていなかった

→ この発見が v0.1.0 の差別化軸（T7 normalizer のスコープ決定）になった。

### 技法 4: 自動分割の挙動

件数超過時の分割処理:

- 分割閾値（本プロジェクト: 法人番号 2000 件、インボイス 500 件）
- 分割番号の取り得る範囲（本プロジェクト: 1-99999 or 1-999999）
- 連番の付番方式（分割跨ぎで連続する？リセット？）

→ `src/lib/pagination.ts` の opaque cursor 変換ロジックが変わる。

### 技法 5: ID ライフサイクル

- 発行リードタイム（本プロジェクト: 2-4 週間）
- 3 年無利用停止条項
- 同一メールアドレスで複数 ID 発行不可条項
- 譲渡・貸与・開示禁止条項

→ Hosted 版の運用設計（Sugukuru がプロキシし個別 ID を渡さない等）が決まる。
